## 테스트 주도 개발 (TDD)
#### TDD (Test Driven Development)
- 매우 짧은 개발 사이클을 반복하는 소프트웨어 개발 프로세스
- 개발자는 먼저 요구사항을 검증하는 자동화된 테스트 케이스를 작성
- 그 이후, 테스트 케이스를 통과하기 위한 최소한의 코드를 생성하고 작성한 코드를 리팩토링하는 과정을 반복

#### 의식 부분
- 일단 간단하고, 해보기 쉬운 것을 먼저 시도
- 실패하는 테스트를 통과하기 위해서는 최소한의 코드를 작성
- 테스트를 점점 구체화할수록 프로덕션 코드는 점점 범용적으로 됨 (커버 가능한 케이스가 많아짐)
- 실패하는 테스트가 있을 때만 프로덕션 코드를 작성
- 실패를 나타내는 데 충분한 정도의 테스트만 작성

#### 장단점
- 작성한 코드가 가지는 불안정성을 개선하여 생산성을 높일 수 있음
- 테스트 가능하면 결합이 느슨한 시스템을 점진적으로 만들어 나갈 수 있음
- 비효율적인 경우도 존재하기 때문에 다른 모든 기술과 마찬가지로 비판적으로 사고하는 것도 중요

---

### 3단계 사이클 (Red -> Green -> Refactor)
#### Red - 실패하는 테스트 작성
- 비즈니스 로직은 비어있거나 미완성 상태
- 테스트가 당연히 실패해야 ㅈ어상
- 이 테스트는 요구사항을 구체적으로 한 단계 더 좁히는 역할

#### Green - 테스트를 통과시키는 최소한의 코드 ㅈ가성
- 여기서 처음으로 프로덕션 코드(비즈니스 로직)를 작성함
- 절대 테스트를 통과시키는 데 필요한 최소한의 구현만 작성

#### Refactor - 코드 정리 및 구조 개선
- 테스트는 이미 초록색이므로 안심하고 리팩토링 가능
- 중복 제거, 이름 개선, 구조 개선 등을 여기서 수행
- 리팩토링 후에도 테스트가 계속 초록이어야 함
